# AI 日报 & 航天周报 · 系统说明

两个自动资讯系统,跑在 **GitHub Actions 云端**,你的电脑关机也照跑。

- **AI 日报**:每天早 8 点,收集 AI 动态,DeepSeek 生成日报,发 163 邮箱
- **航天周报**:每周一早 8 点,收集航天动态,DeepSeek 生成周报,发 163 邮箱

---

## 一、GitHub Actions 运行机制(重点)

### 1. 两个 workflow(各自独立)

| 文件 | 名称 | 频率 | 实体目录 | 窗口 |
|---|---|---|---|---|
| `.github/workflows/daily.yml` | AI 日报 | 每天 北京 8:00 | `entities/`(AI) | 近 3 天 |
| `.github/workflows/space-daily.yml` | 航天周报 | 每周一 北京 8:00 | `entities-space/`(航天) | 近 7 天 |

两个 workflow 互不干扰,各自定时、各自跑、各自发邮件。

### 2. 触发方式

- **schedule(cron 定时)**:到点自动触发。AI 是 `0 0 * * *`(UTC 0:00 = 北京 8:00),航天是 `0 0 * * 1`(每周一)。
- **workflow_dispatch(手动)**:在 GitHub repo 的 Actions 页面点 `Run workflow` 立即触发,测试用。
- ⚠️ Actions 的 cron **不保证准时**,可能延迟几分钟到半小时(高峰期)。别期待精确到分。
- ⚠️ 改了 workflow 文件后,**必须用 `Run workflow` 新触发**,不能用 `re-run`--re-run 跑的是旧代码。

### 3. 运行环境

- `runs-on: ubuntu-latest`:跑在 GitHub 的云端 Ubuntu 服务器(微软 Azure,境外)。
- **境外网络**:reddit / HuggingFace 在本地被墙,在 Actions 上都能通,所以数据比本地全。
- **免费**:public repo Actions 分钟数无限;每天/每周跑一次,零成本。

### 4. 每个 workflow 的运行步骤

```
1. checkout        拉取 repo 代码
2. setup-python    装 Python 3.12
3. pip install openai   装 DeepSeek SDK
4. 设置日期        算北京时区日期(UTC+8)
5. collect.py      收集数据(各源抓实体/关键词) -> data.json
6. report.py       DeepSeek 生成 Markdown 报告(失败则规则化兜底)
7. git push --force   报告 commit 到 repo 的 reports/
8. 发送邮件        163 SMTP 发到你邮箱
```

**collect.py 的关键参数**:
- `--skip`:跳过的源。AI 跳 `x`(X 需 claude CLI);航天跳 `x huggingface arxiv`(X 同前,HF 航天无关,arxiv 关键词搜噪音大)
- `ENTITIES_DIR`:切换实体目录。AI 用默认 `entities/`,航天用 `entities-space/`
- `WINDOW_DAYS`:收集窗口。AI 3 天,航天 7 天

### 5. 关键配置

**Secrets**(repo -> Settings -> Secrets and variables -> Actions):
- `LLM_API_KEY`:DeepSeek API key(`sk-...`)
- `SMTP_PASSWORD`:163 邮箱 SMTP 授权码(不是登录密码)
- Secret 是"只写不读",编辑时 Value 框永远空,填进去 Update 就覆盖

**自动提供**:
- `GITHUB_TOKEN`:Actions 自带,bot 用来 push 报告回 repo(需 `permissions: contents: write`)

**环境变量**(在 workflow 的 `env` 里):
- `ENTITIES_DIR`:实体目录
- `WINDOW_DAYS`:窗口天数

### 6. 数据流

```
cron 到点触发
  → collect.py 从各源抓 80+ 实体动态,打分去重 -> data.json
  → report.py 把 data.json 喂给 DeepSeek 生成中文 Markdown 报告
  → 报告 commit 到 repo reports/ 目录(git push --force)
  → 163 SMTP 把报告发到你邮箱
```

## 二、本地 vs GitHub Actions

| | 本地定时 | GitHub Actions |
|---|---|---|
| 电脑要开机 | 要 | **不要(云端)** |
| reddit / X 被墙 | 是 | 否(境外服务器) |
| 定时稳定 | 本地 cron 7 天过期 | 长期有效 |
| 费用 | 免费 | 免费(public repo) |

当初选 Actions 的原因:不用开机 + 数据全 + 长期稳定。

## 三、数据源

**AI 日报**:`--skip x`,跑 reddit / hackernews / github / huggingface / arxiv

**航天周报**:`--skip x huggingface arxiv`,跑:
- reddit(r/spacex、r/satellites 等按关键词搜)
- hackernews(航天关键词)
- github(航天仓库:NASA、SpaceX-API、Planet Labs 等)
- **知乎**(RSSHub 抓热榜 + 话题,航天关键词过滤,权重 0.6)
- **微信公众号**(RSS,需自配 wechat2rss 等RSS URL,权重 0.7,质量最高)

知乎/公众号配置文件:`entities-space/zhihu_topics.md`、`entities-space/wechat_accounts.md`(每行一个 RSS URL)。

## 四、日常维护

| 想改什么 | 改哪里 | 怎么生效 |
|---|---|---|
| 触发时间/频率 | workflow 的 `cron` | 改完 push |
| 收集窗口 | workflow 的 `WINDOW_DAYS` | 改完 push |
| 跳过的源 | workflow 的 `--skip` | 改完 push |
| 换 LLM | `report.py` / `space_report.py` 的 `base_url` + `model` + Secret `LLM_API_KEY` | 改完 push + 改 Secret |
| 加/删监控实体 | `entities/` 或 `entities-space/` 的 md | 改完 push |
| 配知乎/公众号 RSS | `entities-space/zhihu_topics.md` / `wechat_accounts.md` | 改完 push |
| 查历史报告 | repo 的 `reports/` 目录 或 邮箱 | 直接看 |

**本地改完都要 push 到 GitHub 才生效**(Actions 跑的是 repo 上的代码)。

## 五、故障排查

红叉时点进 run,看**红色那步**:

| 红在哪步 | 原因 | 解决 |
|---|---|---|
| 第一步就红 / YAML 错 | workflow 文件语法/缩进错 | 检查 yml 缩进 |
| 收集数据 | 某源 API 抽风 | 一般有 partial success,不致命 |
| 生成报告,且 `401` | Secret `LLM_API_KEY` 没存对(空/错/带空格) | 重新编辑 Secret,填进去 Update |
| 生成报告,别的错 | DeepSeek 网络/余额 | 看具体错误 |
| 提交报告(push) | 并发 run 冲突 | 用 `git push --force`(已加) |
| 发送邮件 | SMTP 授权码错 / 163 服务问题 | 确认 `SMTP_PASSWORD` 是授权码 |

**本地 push 连不上 github**(443 超时):代理(Clash)没开。开 Clash + `git config --global http.proxy http://127.0.0.1:7890`。

**re-run vs Run workflow**:re-run 用旧代码,改了文件后必须用 `Run workflow` 新触发。

## 七、安全

- GitHub PAT(本地 push 用)用完 revoke 重生成,别长期暴露
- Actions 用自带的 `GITHUB_TOKEN`,不需要你的 PAT
- DeepSeek key / 163 授权码 存 Secrets,不进代码
- 别把 token / key 提交到代码里
